## [2026-07-03T02:32:39.000Z] session 5a774313 — US 18 browser-up: state file vs. stdout

Filed under [[Logbook]]. Design discussion on [[US 18]] (`browser-up`, the browser-lane setup
command) — not yet built, no code changed.

**Question raised:** should `browser-up` persist a local state file recording the running Chrome
(CDP URL + PID) for later verification, instead of just printing the endpoint to stdout?

**Position reached — no, keep stdout, don't add a state file for this story.** Reasoning, all
grounded in the US 18 spec:

- The spec's classify signal is a **live port-reachability probe** ("is a dev-mode Chrome
  answering on the CDP port?"), which is self-correcting. A state file is *stale* truth — it can
  claim "running, PID N" after Chrome died or after the OS recycled the PID, creating two sources
  of truth that disagree.
- The CDP endpoint is **deterministic configured input**, not discovered state: default
  `http://localhost:9222`, overridable via `--cdp`. browser-fetch (US 15/16) attaches using the
  same default/flag, so nothing needs browser-up to hand it a discovered URL. Printing is a human
  convenience, not a handoff channel.
- The PID's only real uses are **kill / health-check** — both explicitly out of scope: AC 6 says
  browser-up leaves Chrome running and never kills a browser it didn't start, and "detecting and
  relaunching a stale/crashed Chrome" is a named **deferred** Later-stage.
- Honest verification is **re-probing the port**, not reading a file.

**Where a state file would earn its place (open thread / future trigger):** when the deferred
stale-Chrome health check gets built, a stored PID becomes the natural discriminator for "the
Chrome *we* launched" vs. one the researcher started — i.e. "is it safe for us to recycle this."
That's the rule 02 "new case → new branch" moment to add it, not now.

**Open thread:** offered to add a one-line DEVLOG deferred flag ("state file (PID + endpoint) is
the natural home for the deferred stale-Chrome health check") so the future trigger is recorded
rather than lost. Awaiting user's go-ahead — not yet written.
