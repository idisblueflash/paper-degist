**Intent.** Decide whether the browser lane needs a dedicated command to *start* a dev-mode
Chrome (vs. the researcher starting it by hand), then draft it.

**Decisions reached.**
- **A dedicated launch command is warranted — as a new US (US 18, `browser-up`).** Reframed from
  the initial lean-toward-no: the operator who starts Chrome during a real run is *Claude Code*,
  not the researcher. The launch incantation (Chrome binary path, flags, profile dir) is knowledge
  Claude re-derives each session, so encoding it once in a command is a direct fit for rule 02
  (consolidate case-handling into the script; each new launch quirk becomes a branch). The
  researcher's role stays the *manual confirmation* (login/captcha), not the launch.
- **It sits one layer before US 15**, and stays launch-only: never logs in, solves walls, or
  fetches (US 15's job). Idempotent — reuse a reachable dev-mode Chrome rather than spawn a second.
- **Two deliberate breaks from the pipeline mold**, both recorded in the US: (1) launch failures
  are a **loud** non-zero exit + diagnostic, NOT a manifest quarantine — there is no paper/batch to
  keep running, so the operator must notice; (2) it does not block waiting on the human — returns
  once the CDP endpoint answers.
- **Fixed persistent `--user-data-dir` (default `.browser-profile/`, overridable)** is the
  mechanism that carries auth across runs → login amortized to the first run, not per run.
  Consequences captured as ACs/deferrals: profile is **secrets-at-rest → must be gitignored**;
  "no manual confirm next time" holds only until **session expiry** (browser-up can't detect it →
  manual re-login); a fixed profile is single-instance (Chrome locks the dir), which reinforces the
  idempotent single-Chrome design and explains the "locked profile" failure branch.
