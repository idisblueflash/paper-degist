**Intent.** User wanted a second parallel working folder on the *same* laptop (not
the mac mini), to carry two branches at once. Asked whether cloning to a sibling
`paper-degist-02` folder is viable, then chose a **full `git clone`** over a
`git worktree`.

**Decisions.**
- **Full clone over worktree.** User picked a full independent clone
  (`../paper-degist-02`, own `.git`) rather than `git worktree` (shared object
  store) — total `.git` isolation between the two checkouts, at the cost of manual
  `git fetch` to sync.
- **Same-machine clone is a *lighter compromise* than rule 09's mac mini, not a
  replacement.** A sibling folder separates the working tree + terminal but **not the
  operator** — still one person, one head, one `git commit` from the wrong tree. Rule
  09's mini remains the stronger separation; the clone is fine when staying on the
  laptop and avoiding SSH round-trips, provided the two folders sit on clearly
  distinct branches.

**Findings worth carrying.**
- **`cp -r` is wrong for this** — it drags the untracked `.venv/` (uv venvs can carry
  absolute paths) and `files/`. Use `git clone` or `git worktree` instead.
- **Per-folder `.venv`** — each checkout needs its own `uv sync`; `uv.lock` is
  committed so versions match.
- **`files/` is untracked** and does not come across a clone/worktree — copy real
  inputs manually for any end-to-end run in the second folder.
- **Latent test bug surfaced:** `test_resolve_oa_cli_missing_email_exits_two`
  (`src/tests/test_cli.py:220`) fails when the shell exports `UNPAYWALL_EMAIL` — the
  CLI reads the env var as the required contact email, so it exits 0 instead of 2. The
  test does not clear the env before `runner.invoke`, so it is **env-leaky**; fails
  identically in the original checkout, not a clone artifact. Candidate DEVLOG deferred
  flag / `monkeypatch.delenv("UNPAYWALL_EMAIL", raising=False)` fix.
